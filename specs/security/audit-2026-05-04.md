**Scope:** OpenSalesTax API at `api.opensalestax.org` and the Astro
demo at `demo.opensalestax.org`, as of v0.54.0.

**Auditor:** Eric Osterberg + Claude Code session (iter-58 follow-up).

This is the first formal security audit for the project. It captures
the baseline so future audits can be diffs against it.

## The one real bug found

`src/opensalestax/app.py` (pre-fix) instantiated `slowapi.Limiter`
with `default_limits=["60/minute"]` and registered the
`RateLimitExceeded` exception handler, but never added
`SlowAPIMiddleware` to the app. slowapi's `default_limits` config is
only applied via that middleware; without it, the limit was inert.

**Repro (live, pre-fix):** 80 back-to-back POSTs to
`https://api.opensalestax.org/v1/calculate` from a single IP all
returned `200`. No `429` ever fired.

**Why it matters:** Public unauthenticated engine. An abuser could
sustain arbitrary RPS, exhausting DB pool / connection capacity /
upstream Cloudflare quotas, with no application-layer brake.

**Fix:** Commit `36e60f7`. Added `app.add_middleware(SlowAPIMiddleware)`
plus a `tests/unit/test_app_security.py` regression that builds a
fresh app with a 3/minute limit and asserts the 4th request gets
`429` -- so the wiring can't silently regress again.

## Defense-in-depth headers added

Same commit added `SecurityHeadersMiddleware`. Every response (200,
400, 401, 429, 500, redirects) now carries:

| Header | Value | Purpose |
|---|---|---|
| `Strict-Transport-Security` | `max-age=31536000; includeSubDomains` | Force HTTPS for one year |
| `X-Content-Type-Options` | `nosniff` | Block MIME-sniffing on JSON responses |
| `X-Frame-Options` | `DENY` | Prevent clickjacking via iframe embedding |
| `Referrer-Policy` | `strict-origin-when-cross-origin` | Don't leak full URLs cross-origin |
| `Permissions-Policy` | `geolocation=(), microphone=(), camera=()` | Disable browser features the API never uses |

**No CSP set.** The `/v1/docs` Swagger UI requires inline scripts
that any practical CSP would block. Cloudflare's defaults plus the
JSON-only API surface cover the realistic browser-side risks.
Self-hosters running behind an nginx fronted by stricter security can
of course add their own CSP at the edge.

## Authentication review

`src/opensalestax/auth.py` -- two modes:

- `open` (default): no auth; per-IP rate limiting only
- `api_key`: `X-API-Key` header required

Key handling:

- 192-bit entropy via `secrets.token_urlsafe(24)`
- SHA-256 hashed in DB; plaintext returned to operator at mint time
  only and never persisted
- `last_used_at` updated best-effort (failures don't break auth)
- Health endpoints exempt from auth in either mode (intentional --
  monitors must always work)
- 401 responses include `WWW-Authenticate: ApiKey`

No findings. Constitution sec 6 / spec sec 6 satisfied.

## SQL / input safety review

- All DB queries go through SQLAlchemy 2 ORM with bound parameters
  (`select(...).where(Model.col == value)`). No string concatenation
  found via grep on `f".*SELECT|f".*INSERT|cursor\.execute\(.*\+`.
- All request bodies validated by Pydantic v2 schemas
  (`src/opensalestax/api/v1/schemas.py`).
- `zip5` and `zip4` are validated as 5-digit / 4-digit strings before
  reaching the lookup engine; non-numeric inputs raise `ValueError`
  in the engine, which the calculate endpoint converts to `400`.
- No HTML output surface anywhere -- responses are exclusively JSON.

No findings.

## CORS review

Default: `OPENSALESTAX_CORS_ALLOWED_ORIGINS=*`.

This is **intentional** for the public `api.opensalestax.org` engine
(the demo and any third-party SPA caller can hit it from any origin)
and is documented in `src/opensalestax/settings.py:71-79`. Self-hosted
deployments should set the env var to a comma-separated allow-list
(e.g. `https://opensalestax.org,https://app.example.com`).

Allowed methods are `GET`, `POST`, `OPTIONS` only.
Allowed headers are `Content-Type` and `X-API-Key` only.

No findings.

## Post-deploy verification (v0.54.1, 2026-05-06)

Built + recreated the api container on `opensalestax-01`, then
confirmed against the live origin (loopback inside the container
namespace):

- 70 parallel POSTs to `/v1/calculate` → **60x 200 + 10x 429**.
  Matches the configured 60/min limit exactly.
- 429 body: `{"detail":"Rate limit exceeded. Slow down or use API-key mode."}`
- All five security headers present on every response (200 and 429
  alike), confirmed via `curl -I` direct against `localhost:8080`
  inside the VM.

A sequential bash `curl` loop at ~1 req/sec does NOT trigger the
limit: the sliding 60-second window ages out old requests fast
enough to keep the bucket pinned at exactly 60. This is correct
behavior for "per-minute" rate limiting -- limits suppress bursts,
not steady-state low-rate usage.

## Recommendations (post-audit)

1. **Trust `CF-Connecting-IP` / `X-Forwarded-For` for real-IP rate
   limiting.** Cloudflare-fronted prod sees the CF edge IP via
   `request.client.host`; multiple edges can each accumulate their
   own bucket, watering down the per-IP limit. Add uvicorn
   `--proxy-headers` plus a slowapi `key_func` that prefers
   `CF-Connecting-IP` over the immediate peer.
2. **Add a Cloudflare Transform Rule** that injects the same five
   security headers on `demo.opensalestax.org` responses. Out-of-band
   from this repo.
3. **Re-run the scan after every minor release.** Add to the
   release checklist in `~/.claude/commands/release.md`.
4. **Tighten CORS** on private deployments. Default is correct for
   the public engine but operators should know to override it.
5. **Add a CSP for `/v1/openapi.json` and `/v1/redoc`** (these don't
   need inline scripts) in a future iteration. Skip `/v1/docs` --
   Swagger UI fundamentally needs unsafe-inline.
