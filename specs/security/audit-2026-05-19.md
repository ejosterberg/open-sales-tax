## Build-only CVEs — compensating controls

The following CVEs are in tools used at build time only (during
`poetry install` / container build), NOT at runtime. They cannot
be triggered by any external request to the production API:

- pip 24.0 (5 CVEs) — bumped to 26.1 via `poetry update`; the
  remaining PYSEC-2026-196 fixes in 26.1.2 (the prod container
  rebuild will pull the latest)
- poetry 1.8.4 (2 CVEs) — host-side tool used by Eric/CI; not
  present in the deployed container
- wheel 0.45.1 (1 CVE) — build tool, same
- dulwich 0.21.7 (2 CVEs) — pulled by poetry for git operations;
  same

**Compensating control:** the container build is single-pass.
Poetry resolves dependencies once with a locked manifest; the
runtime image runs the resolved Python packages with no pip /
poetry / wheel / dulwich present in the runtime layer. An
attacker would need write access to the build environment
(GitHub Actions runner or Eric's laptop) to exploit any of
these — at which point they have already compromised the build
pipeline.

## Conditional CVEs — not applicable to our deployment

### asyncmy 0.2.11 — CVE-2025-65896 (no fix available)

asyncmy is the MySQL/MariaDB async driver. Production deploys
use `postgresql+asyncpg://...` as their DATABASE_URL; asyncmy is
listed in `src/opensalestax/settings.py` only as an example of
the OPTIONAL MariaDB-backend URL format.

**Compensating control:** asyncmy is never imported unless an
operator deliberately sets `DATABASE_URL=mysql+asyncmy://...`.
The CVE is in wire-protocol handling; with asyncmy not loaded,
the attack surface is zero. Future iter could move asyncmy to
a `poetry extras` to make this exclusion explicit (`pip install
opensalestax[mysql]` opt-in).

### pytest 8.4.2 — CVE-2025-71176 (fix in 9.0.3)

CVE describes pytest's predictable `/tmp/pytest-of-{user}/`
directory pattern, exploitable by **local users on the same
machine** for DoS or potentially privilege escalation.

**Compensating control:** pytest runs only in single-user
environments — CI (ephemeral GitHub Actions runner, one user
per job), dev containers (single-user), and Eric's local dev
machine. There is no multi-user shell environment where pytest
runs alongside untrusted users. CVSS scoring confirms local-only
attack vector (AV:L). The pytest 9.x bump is a major release
deferred to a separate PR.

## SonarQube findings — fixed or controlled

### Fixed at source

- 4 MAJOR python:S5799 (implicit string concatenation in
  `src/opensalestax/core/calculate.py`): merged each f-string so
  no `"...a " "b..."` implicit-concat patterns remain. Reduces
  the chance of a silent missing-comma bug.
- 6 CRITICAL python:S1192 (duplicated string literals): replaced
  state-name literals in 5 state modules with `self.state_name`
  references (new_york.py, new_mexico.py, south_carolina.py,
  puerto_rico.py, district_of_columbia.py); extracted
  `"RTA (Collar Counties)"` to a local constant in illinois.py.

### Suppressed with compensating controls (sonar-project.properties)

- 23 CRITICAL python:S3776 (cognitive complexity > 15 on state
  modules + `core/lookup.py` + `core/calculate.py` + `data/
  sst_parser.py` / `zcta_loader.py`): documented in the
  suppression block. Compensating control: high per-state unit-
  test coverage + the 781-row DOR validation grid. Mechanical
  extract-function refactors would scatter the state's tax
  branching across files and harm readability for cross-checking
  against state DOR publications.
